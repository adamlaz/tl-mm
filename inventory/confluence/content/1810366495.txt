# MadPayments Roadmap
# Space: P
# Page ID: 1810366495

Roadmap Objectives Standardize repository branches Share common code Organize for Linux & Android platforms Organize for IDTech VP3350 & Elo M60 hardware profiles The roadmap is split into multiple sequential phases to help organize the code from the “Current Dependency Diagram” to the “Future Dependency Diagram”. Current Libraries and Applications These are libraries and applications as of March 2, 2026 with a synopsis of each. l3-core-lib Root dependency. Branches are organized. Target is the JVM/any particular OS. It does not target any particular hardware profile. madpayments-devices-idtech-neo Needs some branch remediation, but otherwise has all code contained to properly interface with VP3350 via the IDTech Universal SDK. The Linux platform is supported but only linux-x86-64 and linux-aarch64 . Android is not used, nor targeted. macOS is supported by way of darwin-aarch64 but is strictly for development . It is not intended to support ELO. mad-payments-vp3350 Needs some branch remediation. Common code is largely within l3-core-lib . Linux is supported but only platforms and architectures supported by madpayments-devices-idtech-neo . Android is not the intended target platform. macOS is supported but only platforms supported by madpayments-devices-idtech-neo . It is not intended to support ELO. mad-payments-elo-m60 Needs significant changes to support the roadmap in all aspects. It currently serves one target platform: Android. It supports two hardware profiles but does so from different branches . It was originally intended to support ELO but IDTech support was added in a separate branch. New Libraries and Applications These are libraries and applications that should be introduced to support the roadmap. l3-core-android-lib New library repo that contains Android shared platform code. This can safely pull in the Android SDK as a compilation dependency as dependent projects will all be targeting the Android platform. mad-payments-vp3350-android New Android repo that mimics the intent of mad-payments-vp3350 ; isolates Android dependencies and provides a MadPay APK for use specifically with the IDTech Android SDK. The mad-payments-elo-m60 source should be cleaned so no IDTech references remain, purging their dependencies, too. Phase 2: Create New Repositories Prepare to move code where it offers value for Android, ELO, and IDTech Tasks Create the repository l3-core-android-lib Create the repository mad-payments-vp3350-android Create the develop and main (primary) branch in each. Phase 3: Identify and Move Shared Android Code to l3-core-android-lib Extract value from each of the mad-payments-elo-m60 branches that offer it. Ensure the current build of mad-payments-elo-m60 uses the new l3-core-android-lib library on both branches that build to an AAR/APK. Tasks Add dependency l3-core-lib to l3-core-android-lib . Add as a compilation dependency the Android SDK Offer common providers such as logging config preferences (secure storage - Android device vs. PED) NanoHTTPD server (possible to drop and use Ktor?) Remove duplicated Java models/POJOs from mad-payments-elo-m60 . Align serialization with shared core. Publish as com.madmobile:l3-core-android-lib . Phase 4: Identify and Move IDTech Code and Artifacts to mad-payments-vp3350-android Extract from the kiosk branch of mad-payments-elo-m60 all artifacts and code pertaining to the IDTech VP3350 hardware profile. Tasks Add dependency l3-core-android-lib to mad-payments-vp3350-android Deprecate kiosk branch in mad-payments-elo-m60 ; no more commits should be added to it. Simplify mad-payments-elo-m60 ( main branch) to be Elo-only code and artifacts. Phase 5: Standardize All Publish Groups / Namespaces Ensure the group is standardized so CodeArtifact sharing policies can be maintained Tasks Change generic namespace / group com.madmobile as this can lead to cross-contamination by other Bitbucket workspaces. Use a refined sub-group such as com.madmobile.madpayments . Publish to multiple CodeArtifact Maven repositories when cross-workspace builds are concerned. This is particularly important to the Linux MadPay app, as it should be fetched by the Sysco Labs pos-os-upgrades repository to create Debian installers. Phase 6: Build All and Deploy to Devices Ensure both apps can be installed on the same Android device. Tasks Align applicationId strategy for CakePop/Kiosk package queries. Update app manifests ( cake-mobile-pos-v1 , kiosk-v2 ) to query intended MadPay package(s).
