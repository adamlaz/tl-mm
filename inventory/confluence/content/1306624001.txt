Description of Architecture This document outlines a strategy to enhance the CAKE POS device PIN and token retrieval procedure. The architecture aims to streamline device onboarding, minimize support requests concerning PIN and token recovery, empower restaurant operators with self-service capabilities, and modernize authentication methods while preserving security. The solution utilizes current CAKE ecosystem resources and integrates a secure SMS-based token distribution system. This method verifies the operator/installer's identity using a registered mobile number linked to the Cake Merchant account, thus correctly associating the POS device with the merchant. Support agents should follow these steps: Verify the customer's identity following standard authentication protocols Confirm which location is being onboarded (especially important for multi-location merchants) Direct the customer to the existing onboarding screen shown above Generate a new token in Universal Admin/ Super Admin for the specific location Provide both the PIN (if not pre-configured by Ninja One) and the token to the customer The current screen design works well as a fallback solution without requiring modification, allowing for seamless transition between the new SMS flow and traditional support-assisted onboarding. For multi-location merchants, support agents should take special care to verify which specific location's device is being onboarded to ensure the correct PIN and token combination is provided. The UdpMasterBroadcastListenStep on the 2nd POS listens for this before the UI is even loaded. Secured Peer Request: When initializing onboarding, the 2nd POS calls the Master POS's getDetailsFromMaster endpoint with the IP it recieved from UDP. This request is secured using SecurePeerRequestUtil , which signs the request with the existing cake.net certificates. This ensures only authorized devices can fetch the merchant record. Merchant Validation: During the assisted init, the 2nd POS verifies that the provided merchantId and PIN match the details it previously received from the Master POS, providing an extra layer of validation. Assisted Activation: Instead of the user manually entering credentials or using SMS, the 2nd POS uses the merchantId and PIN (fetched from the Master POS) to programmatically obtain an activation token from MAPI via the /assisted endpoint in InitController . CAKEpop & Kiosk Improved Onboarding This improved orderpad/ m60 onboarding process better automates the current manual entry steps taken by an Operator during device setup. PIN/ token assisted entry Use UDP broadcast to securely deliver pin & token. By using a QR code with payload of merchant pin (moxy reads in from N1 env) and token (moxy refreshes token in background), the system eliminates the need for manual entry, ensuring accuracy and saving time. After the user picks a master POS from the LAN list, CakePOP opens a CometD WebSocket to Moxy and shows merchant name, address, and merchant id from messages Moxy already publishes on that channel. The user confirms and taps Activate . CakePOP then calls POST /moxy/rest/init/orderpad/assisted instead of mapi/onboard/information . Moxy uses syncDetails stored from its own earlier onboarding, validates with mapi/orderpad/registry , creates the orderpad Device like the manual flow, and returns Device + syncDetails so CakePOP can finish activation. If Moxy is missing the payment v2 passkey in prefs, it refreshes via mapi/onboard/information before completing. Manual PIN/token entry and Start Over (back to the station list) remain available. classic sequenceDiagram participant Moxy as Moxy master POS CometD participant CakePOP as CakePOP Capacitor
