Building a New Screen from a Prototype with Cursor + Chrome DevTools MCP This guide documents the flow we used to build Security > Audit Log from an external Lovable prototype while keeping the implementation aligned with this repo's presentational split. It is written for engineers and designers using Cursor for agentic development, especially if you have not configured any MCP servers yet. What This Guide Covers using an external prototype as a reference without copying it literally setting up Chrome DevTools MCP in Cursor so the agent can inspect live pages and take screenshots mapping a prototype onto this repo's container + presentational architecture mocking a feature with dummy data before backend contracts exist iterating through design feedback in small, safe passes Prerequisites Cursor Node.js 20.19+ and npm Google Chrome stable this repository checked out locally Set Up Chrome DevTools MCP in Cursor We used Chrome DevTools MCP so the agent could open the Lovable prototype, dismiss the prototype chooser modal, inspect the live page, and take screenshots before implementing anything. Repo Rules to Respect Before You Start This repo is intentionally set up so UI work can happen safely inside presentational boundaries. Safe areas for design-heavy work apps/app/src/components/ui/** apps/app/src/index.css apps/app/src/modules/**/presentational/** Areas to avoid for design-only changes apps/app/src/admin-core/** apps/backend/** apps/app/src/modules/**/queries/** apps/app/src/modules/**/services/** apps/app/src/modules/**/module.ts Preferred architecture route files are containers module-root presentational/ directories hold pure UI routes own loader logic, navigation, search params, query wiring, and mutation wiring presentational files accept props and emit callbacks Useful in-repo references: container example: apps/app/src/modules/menu/routes/items/index.tsx presentational example: apps/app/src/modules/menu/presentational/items/ItemsPage.tsx CRUD presentational example: apps/app/src/modules/kds/presentational/stations/StationsPage.tsx The Workflow 1. Start with the target route and identify the boundary We began with the empty route file at apps/app/src/modules/security/routes/audit.tsx . Before building UI, the first decision was architectural: the route should remain a container, and the screen UI should live under a new module-root presentational directory. For this feature that became: route container: apps/app/src/modules/security/routes/audit.tsx dummy data: apps/app/src/modules/security/routes/audit.data.ts presentational UI: apps/app/src/modules/security/presentational/audit/** That up-front split made later design iteration much easier. 2. Inspect the prototype in a real browser We used the Lovable prototype at: https://staff-and-permissions-prototype.lovable.app/staff/audit-logs With Chrome DevTools MCP available, the agent could: open the page in a browser dismiss the initial prototype chooser modal take a full-page screenshot inspect the visible structure through snapshots summarize the layout hierarchy before writing code This is much better than relying on memory or static assumptions. It let us capture the actual page structure: heading, search/filter toolbar, audit listing, badges, and pagination treatment. 3. Use the prototype for structure, not literal styling The goal was not to clone the Lovable page exactly. Instead, we used it as a reference for: information hierarchy filter controls audit row fields badge usage density of the listing Then we adapted the visual language to our app shell. In practice that meant using existing shared UI primitives and comparing against Menu > Items for spacing and overall page treatment. Good prompt framing for this step: text wide 760 4. Mock with dummy data before backend contracts exist The route did not have live audit APIs yet, so we did not block on backend work. Instead we: created realistic fixture entries in apps/app/src/modules/security/routes/audit.data.ts modeled sources, actions, summaries, and expandable details kept search, filter state, and expanded row state in the route container passed filtered data down into the presentational components as props This let the UI reach a realistic state quickly and created a clear seam for future integration. Good prompt framing for this step: text wide 760 5. Build the first pass quickly The first version intentionally optimized for getting the full flow on screen: page title and helper copy filter controls audit event list expandable details At this stage, speed matters more than perfection. The point is to make the screen reviewable so feedback can become concrete. 6. Iterate in small passes Most of the quality of this screen came from follow-up passes, not the first draft. We iterated through feedback like this: remove summary cards that made the page feel too dashboard-like tighten the page to match existing module pages like Menu > Items remove the oversized outer card shell fix horizontal overflow in the expanded details area add mobile cards instead of forcing the table onto small screens tighten mobile filters into a sticky search bar with two filter buttons This is a good example of how to use Cursor agents well: ask for one meaningful refinement at a time. Good prompt framing for iteration: text wide 760 text wide 760 text wide 760 text wide 760 7. Treat mobile as a first-class pass, not an afterthought The desktop audit table was useful for scanning, but it was not a good mobile pattern. Rather than forcing horizontal scrolling as the main mobile experience, we switched to: desktop: table layout mobile: stacked event cards That split produced a much better small-screen experience while preserving the denser table for larger screens. This is a useful general rule: use tables where density and comparability matter on desktop use cards or stacked rows when the same information must remain readable on phones 8. Validate after each pass After changes, run the narrow checks that matter for this repo: bash wide 760 We used those checks repeatedly while iterating on the feature. Also do a visual pass for: desktop spacing and alignment mobile layout and sticky behavior overflow in expanded rows or filter controls whether the feature still feels consistent with nearby screens in the app Prompt Patterns That Worked Well These were the kinds of instructions that produced the best results: Use the presentational split pattern established in this codebase. Base the design off the prototype, but adapt the design language to match our app. The route should provide dummy data for now. Tighten the page to match Menu > Items. We do not need our content in a card. Focus on mobile view for now. The best prompts combined three things: the architectural boundary the external reference the current refinement goal Common Pitfalls Avoid these mistakes when following this flow: copying the prototype too literally instead of adapting it to the app putting design-heavy code into admin-core mixing route/query logic into presentational files blocking on backend APIs before starting layout work trying to keep the same presentation pattern on both desktop and mobile when it clearly does not fit ignoring overflow bugs inside expandable content A Reusable Template for Future Features You can reuse this sequence for other prototype-driven screens: inspect the existing route and choose the container/presentational boundary open the external prototype with Chrome DevTools MCP take screenshots and summarize the information hierarchy map the design onto our app shell and shared UI primitives mock realistic dummy data if backend work is not ready ship a first presentational pass quickly iterate through focused design feedback one change at a time run check-types and lint:presentational-boundaries Why This Flow Worked This workflow worked well because it kept each concern separate: the prototype gave us structure Chrome DevTools MCP gave us real visual inspection the route container owned state and fixture data the presentational components stayed safe for UI iteration repeated, small design tweaks steadily improved the final result That is the core pattern to reuse: inspect, map, mock, split, iterate, validate.
