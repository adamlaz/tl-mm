1.0 Introduction It has been identified that currently our team members are spending considerable effort on code deployment (on dev, tst and prod environments), troubleshooting environmental issues and managing release artifacts. Majority of these tasks are manual and simple human mistakes lead to wasting of an hours and days of effort. The main objective of this activity is to automate these tasks as much as possible so that teams can focus more on their release deliverables (coding and QE). 2.0 Scope The scope of this document is to identify/describe; How to manage the branches/commits/merges in code repository. How to build the code and bundle the components. How to keep the release artifacts. How to deploy a particular release on an environment (Dev/TST/PROD). How to keep track of deployment histories. Also, the scope of this document is limited to the components where we can override a deployment with a never version (e.g. web applications, apis etc). This document is not considering the components which need to keep the previous state such as databases and messaging queues. 3.0 Assumptions Git/Bitbucket will be used as the code repository. Jenkins will be used as the build and deployment pipelines. All the applications will be containerized and AWS ECR will be used as the container repository. Each git repository will have a ECR repository with the same name. All the applications will be using the config-repo to manage the configurations. AWS ECS will be used to container orchestration and deployment. Required ECS task/service definitions are already configured in the AWS No component will go to an environment without having a release. 4.0 GIT Standards 4.1 Git Branching Standards Git branching standards 4.2 Git Naming conventions Three repositories are required for a new development. 4.2.1. In order to execute the pipeline every repository should have following two scrips in the root directory test.sh → once executed, should be able to run UT for the component. build.sh → once executed should be able to build the component. Dockerfile → once executed should be able to bundle the build artifacts with a docker image. Task <repository name>_task - default task definition <repository name>_<environment name>_task - environment specific task definition Service <repository name>_service Deployment pipeline should check for environment specific task definition is available. If available, it should use the specific task definition for the deployment. If not, it should use the default definition. 10.0 Environment Variables Following environment variables should inject by the deployment pipeline to the images at the time of deployment.
