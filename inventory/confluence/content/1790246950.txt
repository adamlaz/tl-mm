EPIC 1 Title: Salaried Employee Data Model & Effective Dating Description This epic defines and implements the core data model changes required to support salaried employees in CAKE, including persistence, effective dating, salary change handling, and historical reporting integrity. While REST-17995 introduces UI support for salaried employees, this epic ensures that salaried compensation is correctly stored, versioned, migrated, and auditable at the data layer to prevent reporting inconsistencies and payroll inaccuracies. This epic covers foundational backend changes required before full reporting and MLR alignment can be considered complete. Scope 1. Data Model Enhancements Add salaried-specific attributes: Pay Type (Hourly / Salaried) Salary Amount Pay Frequency (Weekly / Annual) Requires Clock-In (Boolean) Ensure backward compatibility for existing hourly employees. Define default values for legacy records. 2. Effective Dating & Change Handling Support effective start date for: Salary creation Salary updates Hourly → Salaried conversion Salaried → Hourly conversion Ensure historical reports reflect compensation structure at time of work. Prevent retroactive corruption of labor records. 3. Salary Change Logic Define behavior for: Mid-period salary changes Multi-day payroll ranges Ensure paid calculations reference correct effective salary. 4. Migration & Backfill Strategy Define migration script for: Existing users defaulting to Hourly Ensure no impact to existing payroll calculations. Validate zero regression on current hourly behavior. 5. Audit & Traceability Log changes to: Pay type Salary amount Frequency Clock-in requirement Ensure compliance and reporting traceability. Acceptance Criteria Salaried attributes persist correctly at the database level. Effective dating prevents historical payroll distortion. Salary updates do not retroactively modify historical labor calculations. Conversion between hourly and salaried does not corrupt prior payroll reports. No regression in hourly workflows. Dependencies REST-17995 (Staff Module UI) REST-18004 (Payroll Reporting) REST-18059 (Labor API Changes) EPIC 2 Title: Core Labor Cost Engine Integration for Salaried Employees Description This epic integrates salaried employees into CAKE’s core labor cost calculation engine beyond the Payroll → Labor Cost report. While REST-18004 updates the Payroll report display, this epic ensures salaried labor is consistently and correctly incorporated into: Labor % Prime Cost Management labor buckets Daily labor summaries Cost aggregations used across reporting and dashboards Without this integration, salaried employees risk being excluded from core operational cost metrics or inconsistently represented across reporting surfaces. Scope 1. Labor Cost Aggregation Logic Define how salaried labor contributes to: Total Labor Cost ($) Management Labor Cost ($) Labor % Prime Cost % Ensure salaried labor is included in daily labor totals. 2. Clock-In Dependency Handling If Requires Clock-In = YES: Use actual worked hours for reporting purposes. If Requires Clock-In = NO: Use salary-based allocation only. Do not fabricate hours. 3. Cost Allocation Rules Define: Daily proration logic (Salary/30 × # of business days or other rule) How partial date ranges are handled Ensure consistency with REST-18004 payroll logic. 4. Management vs Non-Management Classification Determine: Does “Salaried” automatically imply Management labor bucket? Or is classification role-based? Prevent misclassification in labor dashboards. 5. Info Card & KPI Alignment Ensure: Labor info cards reflect salaried costs KPIs remain mathematically consistent No discrepancies between payroll and operational reports Acceptance Criteria Salaried labor appears in total labor cost calculations. Labor % and Prime Cost reflect salaried compensation. No double counting occurs. Hourly labor behavior remains unchanged. Payroll report totals reconcile with labor engine totals. Dependencies REST-17993 (Initiative) REST-18004 (Payroll Reporting) Data Model Epic (above) EPIC 3 Title: Snowflake / MLR Integration & Management Labor Override Alignment Description This epic defines how salaried employees integrate into CAKE Multi-Location Reporting (MLR) and Snowflake labor aggregation pipelines. Currently, REST-17104 introduces a backend-only Daily Management Labor Cost override to compensate for salaried managers not captured via hourly or 7shifts data. With native salaried employee support being introduced under REST-17993, this epic determines: How salaried labor flows into Snowflake. Whether the Daily Management Labor Override remains necessary. Precedence and double-count prevention logic. Enterprise roll-up integrity. This epic is critical to avoid misstatement of labor cost at Location, Region, and Enterprise levels. Scope 1. Snowflake Data Pipeline Updates Extend labor aggregation logic to include salaried employees. Ensure salaried pay is: Allocated per business day Included in management labor metrics where applicable. 2. Override Interaction Rules (REST-17104) Define system behavior when: A location has: Native salaried employees AND A configured Daily Management Labor Override. Determine: Is override additive? Mutually exclusive? Conditional? Prevent double counting. 3. Hierarchy Rollups Ensure salaried labor: Rolls up correctly across: Location Region Enterprise Maintains mathematical consistency across all aggregation layers. 4. Reporting Surface Parity Ensure consistency across: KPI cards Summary rows Sales vs Labor reports CSV / XLSX / PDF / Email exports 5. Backward Compatibility Locations without salaried employees: Continue functioning unchanged. Locations using override only: Continue functioning unchanged unless explicitly migrated. Acceptance Criteria Salaried employees are reflected in Snowflake labor aggregation. No double counting occurs when override is configured. Enterprise rollups remain mathematically correct. MLR results reconcile with Classic labor totals where applicable. REST-17104 behavior is clearly documented as: Temporary workaround OR Long-term configurable feature.
