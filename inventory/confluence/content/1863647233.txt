Crumley Date: March 18, 2026 TL;DR 98.7% of all 502 errors originate from /socket.io . The socket.io server, running with default configuration inside a multi-container ECS deployment, generates a self-perpetuating crash loop. When ECS replaces a container, all connected socket.io clients reconnect simultaneously (thundering herd). Each reconnect triggers an external IDM auth HTTP call inside the socket.io middleware. The burst saturates the Node.js event loop on both ECS tasks simultaneously, causing ALB health checks to fail, both targets to be deregistered, and every subsequent request — including user page loads on browser refresh — to receive a 502 error. ECS then replaces the containers, clients reconnect, and the cycle repeats approximately every 4–6 hours. The Specific Mechanism Step 1 — Transport selection creates HTTP traffic on every reconnect Socket.io is initialized with no options: js wide 760 Socket.io v2.5.1 (the pinned version) defaults to HTTP long-polling as the initial transport before upgrading to WebSocket. Every reconnection attempt — including all reconnects during a storm — starts as an HTTP GET request to /socket.io/?transport=polling . This is why 98.7% of 502 errors appear on the /socket.io path: they are real browser-initiated reconnection requests, not a single client or scanner. Step 2 — Every reconnect fires an external IDM auth call Every new socket.io connection handshake runs through io.use() : js wide 760 idmAuth() is an Express middleware factory that makes external HTTP calls to the IDM service to validate the session cookie on each connection. During a reconnection storm of hundreds of simultaneous clients, hundreds of concurrent outbound HTTP requests are dispatched to IDM within the same second. Step 3 — Node.js event loop saturation Node.js is single-threaded. When hundreds of async IDM callbacks queue simultaneously alongside all other in-flight work, the event loop queue fills. The process becomes unable to respond to new connection attempts. The ALB health check endpoint (an HTTP request to the container) does not receive a response within its timeout window. Step 4 — Both targets deregistered, 502 storm The ALB marks both containers as unhealthy and deregisters them from the target group. With no healthy targets registered, the ALB returns 502 for every incoming request — socket.io reconnects, API calls ( /user/profiles , /mapi/... ), and HTML page load requests from users hitting browser refresh. This is why users see the full AWS 502 gateway error page in their browser window, not just a broken component. Step 5 — ECS replaces containers, cycle resets ECS detects the failed health checks, stops the unhealthy tasks, and launches replacement containers. Once the new containers pass their health checks, they are registered in the target group and the service reaches "steady state." However, because the root configuration issues are unchanged, the next trigger event (a container replacement, a deployment, or accumulated zombie connections) restarts the cycle. Failure Loop wide 760 Contributing Code-Level Issues 1. No transport restriction Without transports: ['websocket'] , socket.io v2 always begins with HTTP polling. In a multi-container ECS deployment without ALB sticky sessions, the client's second polling request can be routed to a different container that has no record of the socket session. The socket is torn down and the client reconnects — increasing baseline reconnection churn even under normal traffic. 2. No sticky sessions on the ALB The socket.io-redis adapter provides pub/sub message broadcasting across containers but does not share per-socket HTTP polling session state . The HTTP polling session is stored in the memory of whichever container handled the initial handshake. ALB round-robin routing means polling requests are frequently routed to the wrong container, causing constant baseline reconnection churn that amplifies the storm when a crash event occurs. 3. No pingInterval / pingTimeout customization Socket.io v2 defaults: pingInterval: 25000ms , pingTimeout: 5000ms . These are not inherently problematic, but without custom tuning, there is no mechanism to shed load (e.g., rejecting connections when the event loop queue depth is high) or to detect zombie connections early. 4. IDM auth middleware on every socket connection The io.use() middleware making an external HTTP call on each connection is the direct trigger for event loop saturation during a reconnection storm. Under normal steady-state traffic this is manageable, but it creates a linear amplification effect: N reconnecting clients → N concurrent IDM HTTP calls → event loop blocked . Root Cause Statement Socket.io 's HTTP long-polling transport, running without transport restriction or ALB sticky sessions in a multi-container ECS deployment, causes elevated baseline reconnection churn. When any trigger event drops connections simultaneously (ECS task replacement, deployment, or a prior crash), the simultaneous reconnection burst dispatches an IDM auth HTTP call per client through the io.use() middleware. The volume of concurrent async callbacks saturates the Node.js event loop on both ECS tasks in parallel, causing ALB health checks to fail, both containers to be deregistered from the target group, and all traffic — including user page loads — to receive 502 errors. ECS automatically replaces the failed containers, clients reconnect, and the cycle repeats.
