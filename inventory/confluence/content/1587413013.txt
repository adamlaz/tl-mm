Introduction KDS v2 is built on a modern, distributed architecture that enables: Multi-station, multi-step kitchen routing Real-time synchronization between devices Strong offline and recovery guarantees Clear separation between order routing logic and execution High performance for busy kitchen environments This page describes the core systems , data flow , and runtime behavior of KDS v2. High-Level Architecture Summary KDS v2 consists of six major components that work together inside every restaurant: 1. Fixed POS Receives and processes orders from Mobile POS, online ordering, and servers Writes all order documents to CouchDB 2. CouchDB Local database on the Fixed POS System of record for all in-store order data Emits a change feed consumed by the Sync Agent 3. Ditto Sync Agent Transforms CouchDB order documents into Ditto documents Publishes Orders and Worksets into the store’s Ditto mesh Handles offline buffering and automatic recovery 4. Router App Produces router-config JSON Defines: Stations (prep and expo) Assembly Lines (routing pipelines, comprised of sequenced stations) Policies (queueing, fallback handling, expo routing) Publishes configuration into Ditto 5. Ditto Mesh Peer-to-peer, multi-protocol database Ensures every KDS tablet has the same state Operates offline and automatically merges changes 6. KDS Client App (Android - Kotlin) Subscribe to Worksets from Ditto Allow kitchen staff to claim, start and finish steps Update Workset state, which syncs instantly to all devices Together, these components provide real-time, distributed coordination across the entire kitchen. Fixed POS Acts as the ingestion point for all orders Ensures orders are written reliably to CouchDB Independent of KDS runtime behavior 2. CouchDB Stores order documents and POS metadata Emits _changes events used by the Sync Agent Provides a consistent base for Workset creation 3. Ditto Sync Agent Transforms and forwards data: Responsibilities include: Subscribing to CouchDB change feed Converting order documents into Ditto Workset format Publishing orders into Ditto collections Currently only converting Couch orders into Worksets Future state will publish full Couch orders over to Ditto. Updating Worksets for order level changes Managing connectivity partitions and resync Ensuring data availability even if POS is offline Its purpose is to keep order data continuously available in the Ditto mesh. 4. Router App Defines and publishes the routing rules: Stations Assembly lines Steps and sequences Prep time and labels Expo routes Queueing and fallback policies The output is a single router-config JSON document that Ditto distributes automatically. 5. Ditto Mesh A distributed peer database inside the store. Key characteristics: Local-first performance Multi-protocol sync (WiFi, LAN, BLE) Conflict-free replication via CRDTs Full offline support Automatic recovery after outages All KDS clients read and write here. 6. Workset Engine Processes orders into Worksets: Builds plan (immutable routing instructions) Initializes state (mutable execution progress) Processes multi-station steps Publishes Worksets back into Ditto This ensures a consistent interpretation of router-config for every order. 7. KDS Clients Each device: Subscribes to its relevant Worksets Filters items by assigned station Allows claiming, starting, completing work Updates the Workset state Resolves changes automatically through Ditto This enables real-time coordination and reduces errors in the kitchen. Architectural Principles Local-First Operation KDS v2 does not rely on the cloud for real-time kitchen operation. Offline Resilience If POS, network, or internet fails: KDS continues uninterrupted All changes sync when connectivity restores Note: offline capabilities need full chaos testing. Obviously, the POS must be functioning to generate an order. We need to test alternate transports for workset to make it to the KDS device is LAN is down. Once Workset is pushed to Ditto mesh, POS, WAN or LAN are not required to work tickets through completion. Separation of Concerns Routing logic in router-config Immutable plan and snapshot in Workset Mutable state driven by KDS tablets Consistent Convergence Ditto’s CRDT engine ensures: No conflicting edits No ordering issues All devices eventually converge Summary KDS v2 is designed for: Reliability — no single point of failure Scalability — supports complex routing and busy kitchens Clarity — Worksets provide structured plan + state Speed — local-first operations ensure fast response times Extensibility — new features and workflows can be added cleanly This architecture provides the foundation for accurate, fast, and highly coordinated kitchen operations across all merchant environments.
