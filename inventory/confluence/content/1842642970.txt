# High Level Architecture
# Space: P
# Page ID: 1842642970

Overview This feature synchronizes Cybersource transaction outcomes from visa-cybersource-transaction-service to Cake's legacy sync endpoint: Endpoint: POST /api/payment_sync/cybersource Purpose: keep Cake payment records aligned with processed Cybersource operations for reconciliation/reporting Design goal: sync is best-effort and must not block primary payment processing High-Level Components 1) Cybersource execution layer File: src/cybersource/cybersource.service.ts Handles provider operations: sale / authorization / credit via process(...) capture void refund incremental reversal After each provider response, builds a sync payload and calls cake integration. 2) Cake integration facade File: src/cake-integration/cake-integration.service.ts Exposes a single method for other modules: syncCybersourceTransaction(merchantId, request) Keeps Cybersource module decoupled from raw transport details. 3) Cake sync API client File: src/cake-integration/cake-sync-api-client.service.ts Owns transport concerns: URL composition custom headers request validation response mapping 4) Sync contract models File: src/cake-integration/model/cybersource-sync.types.ts Defines tagged request/response unions with type as discriminator. Supports operation-specific required fields and optional blocks. End-to-End Flow CybersourceService executes a payment operation against Cybersource. Service receives provider response and emits normal transaction events. Service builds a minimal sync payload aligned to server-side handler requirements. Service calls: cakeIntegrationService.syncCybersourceTransaction(merchantId, payload) CakeSyncApiClientService adds required headers and sends request to Cake sync API. Sync result is logged/observed, but primary provider flow returns normally regardless of sync outcome. Operation Coverage Sync integration is triggered for: sale authorization credit capture void refund incremental reversal Data Mapping Strategy Handler-aligned minimal payloads For each operation, only fields required by the corresponding Cake sync handler are sent. Non-required fields are omitted. Always included identifiers type merchantReferenceId modernPaymentId (always taken from payment.id equivalent in the current flow) Provider transaction id behavior Required operations: sale , authorization , credit , void , refund Optional operations: capture , incremental , reversal If a required field is unavailable, sync is skipped and logged. Authentication and Headers Outbound request headers to Cake: x-email (from config) x-password (from config) x-merchant-account-id (from runtime merchantId argument) Configuration Key config for this integration: CAKE_SYNC_API_HOST - base host for sync API CAKE_SYNC_API_EMAIL - auth email CAKE_SYNC_API_PASSWORD - auth password The client validates required config at startup and fails fast if missing. Error Handling Model Sync is intentionally non-blocking: Sync invocation is wrapped in try/catch Failures are logged only No sync exception is rethrown into payment operation flow Main Cybersource response path remains unchanged
